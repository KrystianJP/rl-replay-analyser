header parsing and full parsing are done asynchronously as header parsing is super fast, while full parsing (using carball) takes multiple seconds, so it allows the replays to be listed immediately while the full parsing occurs in the background. full parsing is serial to each other to give a more accurate progress representation to the user, also it was found to be a about 15% faster than doing it fully asynchronously for some reason.
how this works:
By not using async when defining the get_header function, FastAPI automatically places this task on a separate worker thread
By using it on upload_replay, i start on the event loop, however, by using the anyio.to_thread..., i manually assign the task to run on a worker thread instead of the event loop, to allow the header threads to return their results to the event loop and therefore send their data to the frontend quickly
This would've been done automatically by FastAPI if i excluded the async, however I wanted the uploads to be serial to each other, so I included an upload_lock, which forces each upload request to wait for their turn (this is why starting on the main event loop was required, so i can fully wait for each upload)

After labelling and training a model to classify playstyle, I noticed the accuracy was surprisingly low (72-75%) (and ditched RFC), I theorized this is because of a similar issue I ran into when labelling, where a significant number of players could be classified into 2 different playstyles, so I decided to calculate the probabilities of each playstyle and see the accuracy of the model's top 2 picks instead of just the 1, and got a satisfactory 95.6% accuracy. Therefore, I have decided that a good inclusion to this section would be to include a primary and secondary playstyle if the probabilities of the 2 are close enough, so that the user can see they are a hybrid of 2.
